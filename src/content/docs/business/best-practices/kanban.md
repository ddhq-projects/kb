## Что такое Kanban

**Kanban** — метод управления потоком задач, не привязанный к итерациям. Позволяет визуализировать работу, ограничивать количество задач в работе и плавно улучшать процесс.

Главное отличие от Scrum: Kanban **не требует** изменения текущего процесса — он накладывается поверх существующего.

В контексте современного подхода (Open Guide to Kanban) Kanban — это стратегия оптимизации **потока ценности** через систему, а не только визуальная доска.

## 3 практики Kanban-системы

1. **Определить и визуализировать workflow**
2. **Активно управлять элементами в потоке**
3. **Непрерывно улучшать Flow**

## 4 принципа Kanban

1. **Визуализируй** рабочий процесс — Kanban-доска с колонками
2. **Ограничивай WIP** (Work In Progress) — не больше N задач в каждой колонке
3. **Управляй потоком** — измеряй время, оптимизируй узкие места
4. **Делай правила явными** — Definition of Done, критерии перехода между колонками

## Definition of Workflow (обязательный минимум)

Definition of Workflow — общее явное понимание того, как именно ценность проходит через систему.

Минимальный состав:

1. **Work Items** — какие единицы ценности движутся по потоку.
2. **States** — состояния от "started" до "finished".
3. **WIP Control** — как ограничивается незавершённая работа.
4. **Explicit Policies** — правила входа/выхода и качества переходов.
5. **Service Level Expectation (SLE)** — прогноз времени прохождения с вероятностью.
6. **Визуализация SLE на доске** — чтобы команда управляла рисками старения задач.

## SLE и прогнозируемость

**SLE (Service Level Expectation)** — это рабочее ожидание, например: "85% задач завершаются за 8 дней или быстрее".

SLE помогает:

- прогнозировать поставку с допустимой неопределённостью,
- раньше замечать риски срыва,
- снижать напряжение вокруг срочных задач за счёт прозрачных ожиданий.

## Метрики Kanban

| Метрика | Описание |
|---------|----------|
| **SLE** | Прогноз времени завершения с вероятностью |
| **Work Item Age (WIA)** | Возраст задач, которые уже начаты, но не завершены |
| **Elapsed Time Started-to-Finished** | Календарное время от начала до завершения |
| **Throughput** | Количество завершённых задач за период |
| **Cumulative Flow Diagram** | Визуализация стабильности и узких мест потока |

Дополнительно применяют blocked elapsed time, flow efficiency, flow distribution и time to market.

## WIP-лимиты

WIP (Work In Progress) — ключевой механизм Kanban. Ограничение задач в работе:

- **Выявляет узкие места** — когда колонка заполнена, проблемы становятся видны
- **Уменьшает переключения контекста** — фокус на текущей работе
- **Сокращает Lead Time** — чем меньше задач в очереди, тем быстрее они доходят

Типичные WIP-лимиты: 1-2 задачи на человека в колонке «В работе».

Практическое правило Kanban: **управляй простаивающей работой, а не простаивающими людьми**. Если задачи блокируются, команда фокусируется на разблокировке потока.

## Типичная Kanban-доска

```
Backlog → To Do → In Progress → Review → Done
           WIP:∞   WIP:3        WIP:2     WIP:∞
```

Важно: backlog в Kanban не является обязательным артефактом. Он может формироваться эволюционно по мере работы системы.

## Внедрение Kanban

1. **Визуализируй текущий процесс** — нарисуй доску как есть
2. **Установи WIP-лимиты** — начни с текущего количества задач в работе
3. **Зафиксируй Definition of Workflow** — Work Items, States, Policies, SLE
4. **Управляй потоком** — измеряй WIA, throughput, elapsed time, соблюдение SLE
5. **Введи pull-сигналы** — новое берём только при наличии ёмкости
6. **Улучшай эволюционно** — маленькие эксперименты, не революции

## Когда Kanban подходит

✅ Поддержка и эксплуатации (непредсказуемый поток задач)
✅ Команды с частыми прерываниями
✅ Нет желания ломать текущий процесс
✅ Ops, DevOps, customer support

## Ограничения и риски

- Формальные WIP-лимиты без дисциплины не дают эффекта.
- Частая переприоритизация незавершённых задач старит элементы и ухудшает прогнозируемость.
- Фокус только на output без проверки outcome приводит к росту "псевдопроизводительности".

## Практические сценарии

- Поддержка с SLA: входящий поток инцидентов + прозрачный прогноз по SLE.
- Продуктовая команда с высокой вариативностью задач и зависимостей.
- Внутренние сервисные команды (маркетинг, аналитика, дизайн-операции), где важен баланс срочного и планового.
